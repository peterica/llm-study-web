# Claude Code Project Setup Prompt

> **범용 프로젝트 설정 프롬프트** — 어떤 프로젝트에든 적용 가능한 Claude Code AI 에이전트 환경 구축 가이드.
> 아래 프롬프트를 순서대로 Claude Code에 입력하면, AI가 프로젝트를 분석하고 스킬/에이전트/워크플로우를 자동 구축합니다.

---

## 사용법

1. 새 프로젝트 폴더에서 Claude Code를 실행한다
2. 아래 **Phase 1 ~ Phase 8**을 순서대로 프롬프트에 입력한다
3. 각 Phase의 `{placeholder}`를 실제 프로젝트 내용으로 치환한다
4. AI 응답을 검토하고 승인/수정하며 진행한다

> **Tip**: Phase 1~3 은 한 번에 입력해도 무방. Phase 4~8 은 결과를 확인하며 단계별로 진행 권장.

---

## Phase 1: 프로젝트 기반 설정 (CLAUDE.md)

```
이 프로젝트를 분석하여 CLAUDE.md를 생성해줘. 아래 항목을 포함해야 해:

1. **Project Overview**: LLM 시스템의 전체 흐름(Transformer → Embedding → Retrieval → RAG → Inference → Production)을 이해할 수 있도록, 개념 학습·시스템 맵·인터랙티브 실험을 결합한 LLM 교육 플랫폼을 구축한다. Next.js(App Router) 기반 웹 애플리케이션에 React, Tailwind, MDX 콘텐츠 시스템을 사용하고, System Map 시각화(React Flow/D3)와 인터랙티브 Lab을 통해 LLM 시스템 구조를 체험형으로 학습할 수 있도록 구현한다.
2. **Folder Structure**: 현재 디렉토리를 탐색하여 주요 폴더/파일 트리를 작성
3. **Core Concepts**: 프로젝트의 핵심 개념, 용어, 도메인 지식 정리
4. **Running Commands**: 빌드, 테스트, 린트, 실행 등 주요 명령어 목록
5. **AI Instructions**: AI 에이전트가 반드시 지켜야 할 규칙들:
   - 생성 결과물(빌드 산출물, 생성된 파일)을 직접 수정하지 말고,
     이를 만드는 소스 코드/스크립트를 수정하여 재생성할 것
   - 코드 수정 전 대상 파일을 반드시 Read한 후 Edit할 것
   - 코드 변경 후 반드시 테스트를 실행하여 전체 PASS 확인할 것
   - 명시적 요청 없이 git commit/push하지 말 것
   - 서브 에이전트(Task tool) 병렬 처리 시 동시 최대 3개로 제한할 것
6. **Custom Skills & Agents**: (스킬/에이전트 생성 후 여기에 등록)
```

### 핵심 원칙
- CLAUDE.md는 AI 에이전트의 **진입점** — 세션 시작 시 자동으로 읽힘
- 프로젝트 구조가 변경되면 CLAUDE.md도 반드시 동기화
- 너무 길면 토큰 낭비 → 핵심만 간결하게, 상세 내용은 별도 문서로 분리

## Phase 2: 진행 상황 및 계획 관리

```
다음 두 파일을 생성하고, CLAUDE.md에 세션 시작 시 이 파일들을 먼저 읽도록 지시를 추가해줘:

### PLAN.md
- 프로젝트를 분석하여 개선하면 좋을 작업들을 TODO 체크리스트로 작성
- 우선순위(P0/P1/P2)와 의존 관계를 명시
- 상태 표기: [ ] 대기, [~] 진행중(담당자 표기), [x] 완료, [!] 블로커
- 여러 에이전트가 동시에 작업할 때 충돌을 방지하기 위한 잠금 규칙 포함

### PROGRESS.md
- "진행 중인 작업 (Active)" 테이블: 작업 ID, 내용, 담당 에이전트, 시작 시간
- "최근 세션 요약" 섹션: 날짜, 수행 작업, 변경 파일, 다음 권장 작업
- 세션 종료 시 Active 테이블을 정리하고 세션 요약을 작성하는 절차

### CLAUDE.md에 추가할 규칙:
- 세션 시작 시 PLAN.md와 PROGRESS.md를 먼저 읽고 현재 상태를 보고할 것
- 작업 시작 시 PLAN.md에서 해당 항목을 [~]로 변경하고 담당자를 기록할 것
- 작업 완료 시 [~]→[x], PROGRESS.md에 세션 기록을 추가할 것
- 다른 에이전트가 [~] 상태인 작업은 가져가지 말 것
- 같은 파일을 동시에 수정하지 말 것
```

### 핵심 원칙
- **생성 결과물을 직접 수정하지 않는다** → 결과물을 만드는 소스/스크립트를 수정하고 재생성
- PLAN.md는 "무엇을 할 것인가", PROGRESS.md는 "무엇을 했는가"
- 병렬 작업을 위한 서브 에이전트(Task tool) 활용, **동시 최대 3개**로 컨텍스트 폭발 방지

## Phase 3: 핵심 파일 관리 정책

```
다음 파일들은 git에 커밋하지 않고 로컬에서만 관리한다.
.gitignore에 아래 패턴을 추가해줘:

CLAUDE.md
PLAN.md
PROGRESS.md
.claude/

이 파일들은 AI 에이전트의 작업 환경이며, 프로젝트 코드와 분리하여 관리한다.
작업 후 필요에 따라 업데이트하되, 절대 자동 커밋하지 않는다.
```

> **참고**: 팀원과 AI 설정을 공유하려면 `.gitignore` 대신 별도 브랜치로 관리하거나,
> CLAUDE.md만 커밋하고 PLAN.md/PROGRESS.md는 제외하는 전략도 가능.

## Phase 4: 스킬/에이전트 설계

```
이 프로젝트를 분석하여 필요한 Skills와 Agents 아이디어를 추천해줘.

### 스킬 (Skills) — `.claude/skills/`
반복적으로 수행하는 정형화된 작업. 메인 대화 안에서 지식/워크플로우를 주입한다.
다음 카테고리별로 분류하여 제안해줘:
1. **프로젝트 관리**: 상태 확인, 작업 계획, 세션 종료 처리
2. **파이프라인/빌드**: 빌드, 테스트, 배포, 데이터 생성 등 핵심 워크플로우
3. **품질 검증**: 린트, 테스트, 데이터 무결성, 보안 검사
4. **분석 & 진단**: 코드 분석, 성능 프로파일링, 데이터 탐색
5. **코드 생성**: 보일러플레이트, 템플릿, 스캐폴딩
6. **문서 & 릴리스**: 문서 생성, 변경 이력, 버전 관리
7. **서버 & 배포**: 원격 서버 관리, 데이터 업로드

### 에이전트 (Agents) — `.claude/agents/`
자율적으로 판단하며 수행하는 독립 작업자. 별도 컨텍스트 윈도우에서 격리 실행된다.
스킬과 다른 점:
- description 매칭으로 Claude가 자동 위임 (명시적 호출 없이도 발동)
- 별도 컨텍스트 윈도우에서 독립 실행 (메인 대화 오염 없음)
- 자체 모델, 도구, 권한, 메모리 설정 가능
적합한 작업: 장기 실행, 자동 감사, 패턴 탐지, 회귀 감지 등

각 제안에 대해:
- 이름 (kebab-case)
- 한 줄 설명
- 어떤 상황에서 사용하는지
- 예상 워크플로우 (3~5단계)
```

### 핵심 원칙
- **스킬** = 메인 대화에 주입되는 지식/워크플로우 (`.claude/skills/`)
- **에이전트** = 별도 컨텍스트에서 자율 실행되는 독립 워커 (`.claude/agents/`)
- 너무 많이 만들지 말고, 실제로 자주 쓸 것만 선별

## Phase 5: 스킬 및 에이전트 생성 (품질 기준 준수)

```
이전 단계에서 승인된 스킬과 에이전트를 생성해줘.

---

## Part A: 스킬 생성 (`.claude/skills/`)

### 디렉토리 구조
`.claude/skills/{skill-name}/SKILL.md`

### YAML Frontmatter (필수 필드)
---
name: {skill-name}                    # 디렉토리명과 일치
description: {한 줄 설명}. {트리거 조건}.
argument-hint: "{인자 형식}"           # 선택사항이면 "[선택사항: ...]"
allowed-tools: Read, Grep, Glob, Bash  # 이 스킬이 사용할 도구 목록
---

### 필수 섹션 (이 순서를 지킬 것)

1. **# 제목** — 스킬의 한국어 제목

2. **도입부** — 1~2문장으로 `$ARGUMENTS`를 어떻게 처리하는지 설명

3. **## 실행 시점** — 이 스킬을 실행해야 하는 상황 (3~5개 bullet)
   - 구체적인 트리거 조건 명시
   - "~할 때", "~한 후", "~하기 전" 형태

4. **## 참조** — 관련 파일을 테이블 형식으로 나열
   | File | Purpose |
   |------|---------|
   | `실제/파일/경로` | 이 파일의 역할 |
   - 반드시 `ls`로 검증된 실제 경로만 사용 (플레이스홀더 금지)
   - 프로젝트 소스 파일 + 레퍼런스 문서 모두 포함

5. **## 워크플로우** (또는 ## 실행, ## 단계, ## 체크리스트)
   - `### Step N: 단계명` 형식으로 순서대로 기술
   - 검증 스킬의 경우 각 단계에 **PASS/FAIL 기준** 포함:
     - PASS: {통과 조건}
     - FAIL: {실패 조건과 대응 방법}
   - 사용할 도구와 명령어를 명시

6. **## 결과 출력** (또는 ## 보고) — 출력 형식 (마크다운 테이블 등)

7. **## 예외사항** — "다음은 **문제가 아닙니다**:" 형식으로 2~3개
   - False positive 방지를 위한 면제 케이스
   - 프로젝트 도메인에 맞는 현실적인 예외

### 스킬 템플릿

---
name: {skill-name}
description: {한 줄 설명}. {트리거 조건 요약}.
argument-hint: "{인자 형식}"
allowed-tools: Read, Grep, Glob, Bash
---

# {스킬 한국어 제목}

`$ARGUMENTS`를 {처리 방식}.

## 실행 시점

- {트리거 조건 1}할 때
- {트리거 조건 2}한 후
- {트리거 조건 3}하기 전

## 참조

| File | Purpose |
|------|---------|
| `{실제/파일/경로}` | {이 파일의 역할} |

## 워크플로우

### 1. {첫 번째 단계}

{단계 설명}

- **PASS**: {통과 조건}
- **FAIL**: {실패 조건}

### 2. {두 번째 단계}

...

## 결과 출력

| 항목 | 결과 |
|------|------|
| ... | ... |

## 예외사항

다음은 **문제가 아닙니다**:

1. **{예외 제목}** — {왜 문제가 아닌지 설명}
2. **{예외 제목}** — {왜 문제가 아닌지 설명}

## Part B: 에이전트 생성 (`.claude/agents/`)

에이전트는 스킬과 다른 시스템이다. 별도 컨텍스트 윈도우에서 격리 실행되며,
Claude가 description을 기반으로 자동 위임한다.

### 파일 구조
`.claude/agents/{agent-name}.md` (단일 마크다운 파일, 폴더 아님)

### YAML Frontmatter

---
name: {agent-name}
description: {자동 위임 트리거 조건을 포함한 구체적 설명}. {어떤 상황에서 사용하는지}.
tools: Read, Grep, Glob, Bash
model: inherit                         # 선택: haiku(경량), sonnet(기본), opus(고품질)
memory: project                        # 선택: 세션 간 학습 보존 (user/project/local)
---

**필수 필드**:
- `name`: 파일명과 일치 (kebab-case, 예: `my-agent` → `my-agent.md`)
- `description`: Claude가 자동 위임 여부를 판단하는 기준. 구체적으로 작성할 것
- `tools`: 에이전트가 사용할 도구 (쉼표 구분, 예: Read, Grep, Glob, Bash)

**선택 필드** (필요한 것만 사용):
- `model`: 에이전트 실행 모델 (기본: inherit = 부모 모델 상속)
  - `haiku`: 경량 작업 (탐색, 검색, 간단한 분석)
  - `sonnet`: 범용 (기본)
  - `opus`: 고품질 판단이 필요한 복잡한 작업
- `memory`: 세션 간 영구 메모리 범위
  - `project`: 프로젝트별 학습 보존 (`.claude/agent-memory/`)
  - `user`: 전체 프로젝트 공통 (`~/.claude/agent-memory/`)
  - `local`: 프로젝트별, git 미추적 (`.claude/agent-memory-local/`)
- `background`: true로 설정하면 백그라운드에서 비동기 실행
- `isolation`: worktree로 설정하면 별도 git worktree에서 안전하게 작업
- `maxTurns`: 최대 실행 턴 수 제한
- `skills`: 에이전트에 주입할 스킬 이름 배열

### 에이전트 본문 작성 규칙

마크다운 본문은 에이전트의 **시스템 프롬프트**가 된다.
에이전트는 메인 대화의 시스템 프롬프트를 받지 않으므로,
본문에 필요한 모든 지시사항을 포함해야 한다.

### 에이전트 템플릿

---
name: {agent-name}
description: {구체적 설명 — Claude가 이 설명을 보고 자동 위임 여부를 결정한다}.
tools: Read, Grep, Glob, Bash
model: inherit
---

# {에이전트 한국어 제목}

{에이전트의 목적과 자율 수행 범위}.

## 실행 시점

- {트리거 조건 1}
- {트리거 조건 2}
- {트리거 조건 3}

## 참조

| File | Purpose |
|------|---------|
| `{실제/파일/경로}` | {역할} |

## 워크플로우

### 1. 분석 (Analysis)
{현재 상태를 파악하는 단계}

### 2. 계획 (Planning)
{수행할 작업 목록을 도출하는 단계}

### 3. 실행 (Execution)
{실제 작업을 수행하는 단계}

### 4. 검증 (Validation)
{결과를 검증하는 단계}

### 5. 보고 (Report)
{결과를 사용자에게 보고하는 단계}

## 예외사항

다음은 **문제가 아닙니다**:

1. **{예외}** — {설명}
2. **{예외}** — {설명}

### description 작성 가이드

description은 Claude가 자동 위임 여부를 결정하는 핵심이다.
구체적이고 명확하게 작성할수록 의도한 상황에서만 정확히 발동한다.

좋은 예:
- "전체 데이터셋 품질을 자동 점검합니다 — 샘플 수, 중복, 슬롯 커버리지, 문법 이슈를 종합 분석합니다."
- "코드 변경에 대한 테스트를 자동 생성합니다 — 변경된 함수를 분석하여 누락된 테스트 케이스를 추가합니다."

나쁜 예:
- "품질 점검" (너무 모호 — 어떤 품질? 어떻게?)
- "테스트 생성기" (언제 발동해야 하는지 불명확)

### 스킬 품질 기준 (모든 스킬이 충족해야 함)
- [ ] `name` 필드가 frontmatter에 존재하고 디렉토리명과 일치
- [ ] `## 실행 시점` 섹션에 3개 이상의 구체적 트리거 조건
- [ ] `## 참조` 섹션이 `| File | Purpose |` 테이블 형식
- [ ] 참조의 모든 파일 경로가 `ls`로 검증된 실제 경로
- [ ] 검증 스킬의 워크플로우에 PASS/FAIL 기준 명시
- [ ] `## 예외사항` 섹션에 2개 이상의 현실적인 면제 케이스
- [ ] 일관된 마크다운 형식 (닫히지 않은 코드 블록 없음, 테이블 열 일치)

### 에이전트 품질 기준 (모든 에이전트가 충족해야 함)
- [ ] `name` 필드가 frontmatter에 존재하고 파일명과 일치 (예: `my-agent` → `my-agent.md`)
- [ ] `description` 필드가 자동 위임 트리거 조건을 포함하여 구체적으로 작성됨
- [ ] `tools` 필드가 쉼표 구분 형식으로 존재 (예: Read, Grep, Glob, Bash)
- [ ] `## 실행 시점` 섹션에 3개 이상의 구체적 트리거 조건
- [ ] `## 참조` 섹션이 `| File | Purpose |` 테이블 형식
- [ ] 워크플로우가 5단계 (분석→계획→실행→검증→보고) 구조
- [ ] `## 예외사항` 섹션에 2개 이상의 현실적인 면제 케이스
- [ ] 본문에 에이전트가 자율 수행하는 데 필요한 모든 지시사항 포함

생성 후 CLAUDE.md의 "Custom Skills & Agents" 섹션에 스킬/에이전트 목록 테이블을 추가해줘.
스킬은 `.claude/skills/` 하위, 에이전트는 `.claude/agents/` 하위에 각각 정리.
```

```
스킬과 에이전트가 공유할 참조 문서를 `.claude/references/` 디렉토리에 생성해줘.

### 필수 레퍼런스
1. **project-conventions.md** — 프로젝트 규칙, 코딩 컨벤션, 네이밍 규칙, 환경 설정

### 규칙

- 레퍼런스 문서가 변경되면 참조하는 스킬/에이전트도 함께 업데이트

```

## Phase 7: 검증 및 유지보수 루프

```
스킬/에이전트 시스템의 자기 유지보수를 위한 2개의 메타 스킬을 생성해줘:

### 1. verify-implementation (통합 검증 오케스트레이터)
- 프로젝트에 등록된 모든 `verify-*` 스킬을 순차 실행
- 각 스킬의 Workflow에 정의된 검사를 실행하고 PASS/FAIL 판정
- 각 스킬의 예외사항을 참조하여 false positive 방지
- 발견된 이슈에 수정 방법을 제시하고, 사용자 승인 후 수정 적용
- 수정 후 재검증으로 Before/After 비교

### 2. manage-skills (스킬/에이전트 유지보수)
- 현재 세션의 git 변경사항을 분석
- 변경된 파일이 기존 스킬(`.claude/skills/`) 및 에이전트(`.claude/agents/`)의
  커버리지에 포함되는지 매핑
- 커버되지 않는 변경 → 기존 스킬/에이전트 UPDATE 또는 새로 CREATE 제안
- 스킬 생성/수정 시 스킬/에이전트 생성 단계에서 정의된 품질 기준을 강제 적용 (스킬/에이전트 각각의 기준)
- 생성 후 CLAUDE.md, verify-implementation, manage-skills 자체를 자동 업데이트

### 유지보수 트리거
- 새 기능 구현 후
- PR 생성 전
- 주기적 정기 점검
```

## Phase 8: 자율 반복 워크플로우

```
AI가 반복적으로 작업을 수행할 수 있는 자율 워크플로우 스킬을 생성해줘:

### auto-work (자율 작업 루프)
1. PLAN.md에서 미완료 작업 중 우선순위가 가장 높은 것을 선택
2. 해당 작업에 적합한 스킬/에이전트가 있으면 호출, 없으면 직접 수행
3. 작업 완료 후 관련 테스트/검증을 실행
4. PLAN.md 업데이트 ([~]→[x]) 및 PROGRESS.md에 기록
5. 다음 작업이 있으면 1로 돌아감 (--max-tasks N으로 반복 횟수 제한)

### 실행 플로우
PLAN.md 읽기 → 작업 선택 → 실행 → 검증 → 기록 → (반복)

### 안전장치
- `--max-tasks N`: 최대 N개 작업 후 중지 (기본 3)
- 테스트/검증 실패 시 해당 작업을 [!] 상태로 표시하고 다음 작업으로
- 에이전트 실행은 병렬 최대 3개로 제한
- 각 반복마다 사용자 확인이 필요하면 --interactive 플래그
```

## 전체 실행 순서 요약

```
Phase 1 → CLAUDE.md 생성 (프로젝트 기반)

Phase 8 → 자율 워크플로우 실행 (자동화)
```

## Appendix C: 스킬 vs 에이전트 선택 가이드

어떤 작업을 스킬로 만들지, 에이전트로 만들지 판단 기준:

| 기준 | 스킬 (`.claude/skills/`) | 에이전트 (`.claude/agents/`) |
|------|--------------------------|------------------------------|
| **실행 환경** | 메인 대화에 지식 주입 | 별도 컨텍스트에서 격리 실행 |
| **호출 방식** | `/skill-name` 명시적 호출 | 자동 위임 + `/agent-name` 호출 |
| **자율성** | 낮음 — 정해진 워크플로우 실행 | 높음 — 스스로 판단하며 수행 |
| **실행 시간** | 짧음 (1~5분) | 길 수 있음 (5~30분) |
| **컨텍스트** | 메인 대화 컨텍스트 공유 | 독립 컨텍스트 (메인 오염 없음) |
| **메모리** | 세션 내에서만 유효 | 세션 간 영구 보존 가능 |
| **모델** | 부모 모델 고정 | 별도 모델 선택 가능 |
| **적합한 작업** | 검증, 빌드, 배포 체크리스트, 문서 생성 | 품질 감사, 패턴 탐지, 회귀 감지, 테스트 생성 |
